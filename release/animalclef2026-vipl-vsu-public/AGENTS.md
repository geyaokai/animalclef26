# Repository Guidelines

## Project Structure & Working Areas
This is a Kaggle competition workspace. Official files live at the root: `metadata.csv`, `sample_submission.csv`, `animal-clef-2026.zip`, and `images/`. Use `metadata.csv` as the source of truth for IDs, paths, splits, datasets, and identities.

`doc/` is the main planning and communication area and should stay mostly in Chinese. Use `doc/01_计划与沟通/01_沟通与计划.md` for task tracking, `doc/02_数据与预处理/10_数据概览与任务要求.md` and `doc/02_数据与预处理/20_数据预处理.md` for data notes and preprocessing decisions, `doc/03_实验与结论/30_模型训练与评测.md`, `doc/03_实验与结论/40_训练与推理技巧.md`, and `doc/03_实验与结论/60_关键结论与结果归档.md` for experiments and durable findings, `doc/04_算法与代码解释/` for split Feynman-style explanations of key code paths, algorithms, tensor or array shapes, metrics, and abbreviations, and `doc/05_规范与验证/50_Codex测试与验证约束.md` for validation rules.

`example/` stores third-party reference code and sourced research notes only. Keep one example per subdirectory with source URL, version or commit, and license notes. Imported reports must also record their online source. If project code is added later, place it in `src/`, `tests/`, `notebooks/`, and `submissions/`.

Current local code lives in `src/animalclef_analysis/` and `scripts/`. Generated analysis artifacts should go to `artifacts/`, with the first-round audit stored in `artifacts/initial_audit/`.
Reversible preprocessing baselines live under `artifacts/preprocessing_baselines/v1/`, including salamander orientation-normalized images, duplicate flags, and training/test manifests.
Sampled SAM segmentation probes live under `artifacts/sam3_probe/v1/`, with overlays, masked crops, per-image tables, and qualitative contact sheets.
ORB rerank validation artifacts live under `artifacts/descriptor_baselines/orb_rerank_v1/`, including smoke-test tables, dataset gate decisions, local match score tables, reranked validation predictions, and top-1 transition qualitative sheets.
Training artifacts are being standardized under `artifacts/training/`. New runs should prefer:
- `artifacts/training/experiments/<experiment_id>/` for formal runs
- `artifacts/training/cache/<cache_id>/` for teacher caches
- `artifacts/training/debug/<probe_or_smoke_id>/` for short probes
If an older run still lives directly under `artifacts/training/<run_id>/`, do not rewrite it just for cleanup; follow the standardized layout for new work.
Submission-ready inference outputs should go under `artifacts/submissions/<submission_id>/`, including `submission.csv`, per-dataset prediction tables, route config, and a short summary report.
Every formal submission summary under `artifacts/submissions/<submission_id>/reports/summary.md` must include an explicit architecture section. At minimum, state the overall system type, the per-dataset branch or route, the backbone or descriptor name, the key tensor or embedding dimensions, whether rerank or post-processing is enabled, and the clustering threshold used for each dataset. Do not leave official submissions as anonymous csv drops.

## Data & Task Notes
AnimalCLEF 2026 is an unsupervised clustering task scored by Adjusted Rand Index (ARI). The local export includes `LynxID2025`, `SalamanderID2025`, `SeaTurtleID2022`, and `TexasHornedLizards`; if website summaries differ, follow local `metadata.csv` and `sample_submission.csv`. Submission files must preserve `image_id,cluster`. External data is allowed, but keep an official-data-only comparison run. AnimalCLEF 2025 materials are useful background for representation learning and preprocessing, but their open-set thresholding and database/query submission logic must not be copied into the 2026 pipeline without explicit local validation.
Kaggle submission budget is currently capped at `5` submissions per day. Treat this as a hard planning constraint, not an afterthought. Default policy:
- do not spend a submission on a change that cannot be described as a single primary factor relative to the current best official route
- prefer `1` anchor or baseline-preserving submission, `2` high-confidence variants, `1` exploratory variant, and keep `1` slot in reserve until late in the day unless the user explicitly wants an aggressive sweep
- before each official submission, record the baseline being compared against, the only intended change, the reason it deserves one of the `5` daily slots, and the expected risk in `doc/01_计划与沟通/01_沟通与计划.md`
- if a variant is mainly engineering solidification of an already-submitted route, do not consume an extra slot unless the user explicitly wants official confirmation
Current preprocessing evidence is dataset-specific: skip SAM for `LynxID2025`, prioritize optional SAM branches for `SalamanderID2025` and `TexasHornedLizards`, and treat `SeaTurtleID2022` segmentation as fallback-only unless a more reliable ROI strategy is validated.
Current rerank evidence is also dataset-specific: keep plain `fusion` for `LynxID2025` and `SeaTurtleID2022`, and only enable the current ORB local-match rerank branch for `SalamanderID2025` unless a stronger local matcher proves otherwise.
When the daily Kaggle submission quota is exhausted, the CLI may return a generic `400 Bad Request` even after the file upload progress finishes. If the csv schema matches `sample_submission.csv` and `kaggle competitions submissions -c animal-clef-2026` shows no new row, treat that `400` as quota exhaustion first rather than as a networking or csv-format issue.
Data preview work must include identity-count distribution plots derived from ground truth for the labeled training data: overall distribution across all labeled training identities, plus per-dataset views when useful, with explicit reporting of how many identities have only one image. When a baseline uses a derived split such as train/valid holdout, also include the split-specific identity-count distribution in the data preview or experiment notes so the user can see what the baseline actually trained and validated on.
Current supervised-training priority is:
1. `ft_miew_arcface_distill_v1`
2. `ft_mega_arcface_distill_v1`
3. `ft_convnext_arcface_distill_v1`
4. `ft_miew_arcface_masked_supcon_v1`
5. `ft_salamander_subcenter_v1`
This order replaces older placeholder plans where `ConvNeXt` was treated as the only default fine-tuning mainline. `ConvNeXt` is now a control or student baseline, while `MiewID` and `MegaDescriptor-L` initialized fine-tuning are the primary performance-oriented runs.
Formal docs must track performance-improving tricks explicitly. Keep `doc/03_实验与结论/40_训练与推理技巧.md` updated with a trick table that records the baseline, the modified method, the evidence scope (`local val`, `public LB`, later `private LB` if known), the before/after metric, the delta, and whether the trick is retained. Important gains must not live only in chat or only inside one submission summary.

## Coding Style & Naming Conventions
Use Python with 4-space indentation, type hints where practical, `snake_case` for modules and functions, and `PascalCase` for classes. Keep filenames descriptive, for example `src/embedding_pipeline.py` or `submissions/ari_baseline_v1.csv`. Do not rename official columns, dataset IDs, or image paths. Important decisions and stable conclusions belong in `doc/`, not only in chat. When summarizing external papers or adapting third-party code, record the source URL and the local path where it was stored. When explaining implementation details, always include key tensor or array dimensions and note shape changes at each important step. Use a Feynman-style explanation for algorithm or code walkthroughs: explain what problem a step solves, what the input is, what the output becomes, and why the step exists. Do not let explanation files grow without bound: split them by topic under `doc/04_算法与代码解释/` when a file starts mixing multiple algorithms or concepts. In summaries and reports, expand important abbreviations and briefly explain critical terms the first time they appear. If a summary or report depends on figures, reference the figure images directly inside the markdown instead of only saving them beside the report, and add a short explanation of how to read each figure: what the axes mean, whether scales are linear or log, what comparison the user should make first, and what takeaway matters. Report experimental work in a paper-like concise style: always surface key setup details that materially affect interpretation, such as which data subset was used, how train/valid were split, what threshold range was scanned, and what the main comparison result was, without forcing the user to inspect code. Do not dump every low-level implementation detail by default; reserve deeper mechanics for the dedicated explanation docs or for follow-up questions.
Supervised training runs should save local monitoring plots under `reports/plots/` in addition to raw csv logs. At minimum, include training loss curves, validation metric curves, and teacher-student alignment curves when distillation is used, and embed those figures back into `reports/summary.md` with short reading notes.
When changing supervised training batch size, also record the effective batch size and rescale learning rates accordingly instead of silently reusing old LR values. Save both the reference LR and the resolved LR after scaling in the config and summary.
Do not blindly reuse a previous probe result for a new experiment. A batch-size probe can be reused only when the student backbone, input resolution, AMP mode, loss branches, per-device batch logic, and `world_size` are unchanged; otherwise run a fresh probe.
On the current `24 GB` cards, treat batch probing as a search for a stable steady-state memory band rather than a single lucky batch. As a default heuristic, aim for roughly `70%` to `85%` peak memory usage on a formal run; if the peak is far below that band, test a larger batch, and if it is near `90%` or unstable, back off.
Training `summary.md` files are experiment cards, not raw logs. They must at least expose the experiment goal, split protocol, fit/val scale, model/init/head structure, resize and augmentation, per-device batch size, gradient accumulation, effective batch size, optimizer, base and resolved learning rates, weight decay, scheduler with warmup, epochs, main loss weights, teacher sources and dimensions, best metrics, peak CUDA memory, monitoring figures, qualitative review, and the next-step judgment.
Checkpoint policy: preserve the full `model_state_dict`, which includes dataset-specific `ArcFace` or `Sub-center ArcFace` heads, and record head shapes in a human-readable place such as `summary.md` or a sidecar json. Prefer saving optimizer, scheduler, and AMP scaler state as well for resumability.
Long-running GPU work should use `tmux` when practical. Parallel and multi-GPU runs are allowed only when there is clear free capacity. Do not kill, reset, or interrupt unrelated user or system processes merely to free GPU memory; only manage sessions and processes started for the current experiment.
Long-running network-bound preprocessing or model-download jobs should also use `tmux`, not only training jobs, so partial progress can survive transient SSH or CLI interruptions.
Before each formal training run, do a batch-size probe on the intended backbone and input size using the real training loop. Check `nvidia-smi`, record peak CUDA memory, decide whether to keep that batch size or back off, and then decide whether the machine is better used for parallel single-GPU experiments or a multi-GPU run.
Default scheduling priority is: single-GPU probe first, then parallel single-GPU formal runs for independent experiments, and only then multi-GPU for one experiment if single-card batch has already been pushed to a justified limit or wall-clock time is the true bottleneck. Any multi-GPU result must record `world_size`, per-device batch, gradient accumulation, effective batch size, and LR scaling before it is considered comparable. Even single-GPU runs should record the resource snapshot and why that device was chosen.

## Testing Guidelines
There is no automated suite yet. Add `pytest` smoke tests under `tests/` when code appears. Minimum validation is: schema, row-count, and path checks for data; a small smoke test before full training; and row count, column names, and `image_id` order checks against `sample_submission.csv` for inference. Subjective evaluation is required when models are added: save hard-error examples, nearest-neighbor visualizations, and random cluster samples, then reference them from `doc/03_实验与结论/30_模型训练与评测.md`. Any baseline or experiment report must explicitly state the split protocol used for evaluation, including whether the split is by image or by identity, the holdout fraction, the random seed, and which datasets participated in validation. Any logic adapted from `example/` must note the source and how it was changed for the 2026 clustering task. Codex should not mark work complete without running relevant checks or explicitly stating what could not be verified.

## Commit & Pull Request Guidelines
No local `.git` history is present, so there is no project-specific style to mirror. Use short imperative subjects such as `feat: add ARI clustering baseline` or `fix: preserve sample submission order`. Pull requests should list data used, external data usage, validation commands, and the related notes under `doc/`.
