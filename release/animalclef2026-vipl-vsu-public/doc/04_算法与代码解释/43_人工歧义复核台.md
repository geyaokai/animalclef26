# 人工歧义复核台

## 这个工具和论文思路像在哪里

- 那篇 AAS 论文的核心不是“盲目多标”，而是先找 **ambiguity zone**，再把人的精力放到最有信息量的模糊样本对上。
- 我们现在这个工具借的是这个思路，但不照搬“主动学习重训练”：
  - 论文里：`双聚类分歧 -> 人工标注样本对 -> 约束聚类 / 再训练`
  - 我们这里：`多聚类分歧 -> 人工审核 pair -> 导出 pair judgments -> 后续再生成 split/merge`
- 也就是说，它现在首先是一个 **pair-first ambiguity review workbench**。

## 这个工具现在能做什么

- 加载 base submission 的 `test_predictions_v1.csv`
- 加载 ambiguity probe 的：
  - `test_split_candidates_v1.csv`
  - `test_merge_candidates_v1.csv`
  - `test_pair_disagreement_v1.csv`
- 在界面里看：
  - 候选簇
  - 支撑边
  - 成员图像
  - pair 的 `sim / ambiguity_score / votes`
- 在界面里只做一件事：
  - 对当前 pair 打 `yes / no / uncertain`
- 导出成 `pair_judgments.json`
- 后面再把这批 judgments 编译成 `split / merge / attach`

## 现在的入口

- 交互脚本：`scripts/launch_manual_review_workbench.py`
- 后端逻辑：`src/animalclef_analysis/manual_review_workbench.py`
- pair judgment 导出器：复核台内置导出
- overlay 执行器：`scripts/build_manual_cluster_overlay_submission.py`（后续阶段再用）

## 启动命令

## 界面分成哪几块

### 1. Pair Review

- 先看 candidate task 面板，不再靠下拉框找候选
- 每一行都会直接显示：
  - `待处理 / 进行中 / 已完成`
  - `已判 / 总 pair`
  - `yes / no / uncertain`
  - 当前候选的数据集、规模、优先分
- 点任务行，就会加载对应的 `split` 或 `merge` candidate
- 也可以点 `下一条未完成任务`
- 看候选 summary
- 看成员图
- 看 support pair
- 对 pair 打：
  - `同一个体` = `yes`
  - `不是同一个体` = `no`
  - `不确定` = `uncertain`
- 加入 judgment 队列
- 每次点击后会自动跳到：
  - 当前任务里的下一对未判 pair
  - 如果当前任务做完，就自动切到下一条未完成任务

这块最像论文里的 **ambiguity-aware sampling pool**，只是我们不是去采样训练标注，而是采样“pair judgment”。

### 2. Judgments / Export

- 看当前已经积累的 pair judgment
- 可以删除选中、删除最后一个、整批清空
- 导出 `pair_judgments.json`
- 这些 judgment 之后再转成具体 `split / merge`

## 为什么任务式界面更适合现在

- 之前下拉框的问题是：
  - 候选一多，根本不知道自己看到第几个
  - 也不知道哪些已经看过、哪些没看过
  - 更像“找条目”，不像“做任务”
- 现在改成任务表以后：
  - 一眼就知道还有多少 `pending`
  - 哪些 `in_progress` 应该优先收尾
  - 哪些已经 `completed`
- 所以它更适合你后面连续人工扫 `TexasHornedLizards` 或更大批次的 test ambiguity 区

## 为什么这特别适合 Texas

- `TexasHornedLizards` 目前最大的问题不是“完全没路”，而是：
  - 没有本地标签校准
  - 一些 cluster 视觉上很容易看出错并/误拆
  - 但直接大改主路风险很高
- 所以最自然的策略就是：
  - 先靠 cached route 或 cluster 形态找可疑区
  - 在复核台里少量人工确认 pair
  - 先积累 `must-link / cannot-link` 证据
  - 再决定要不要生成小规模 overlay 包去试 official

## 当前边界

- 现在更强的是“先收集 pair 证据”，不是直接在 UI 里硬做最终聚类
- 目前还没有把“图染色 + 匈牙利匹配”那套约束聚类直接做进 UI
- 但这一步已经把最关键的人机闭环打通了：
  - 看图
  - 判 pair
  - 导出 judgments
  - 再做后续聚类修补

## 缓存与恢复

- 现在 judgment 会自动缓存到：
  - `artifacts/analysis/manual_review_sessions/autosave/<session>.json`
- 每次你点：
  - `同一个体`
  - `不是同一个体`
  - `不确定`
  - 删除 judgment
  - 清空 judgment
  都会自动把最新状态写回缓存
- 页面刷新后，工作台会自动恢复最近一次 session
- 所以正常情况下，不需要再担心“手滑刷新一下就全没了”
